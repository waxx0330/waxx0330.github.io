这篇文章记录我对 Agentic RL 的初步理解。相比普通的语言模型训练，Agentic RL 更关注一个问题：**当模型不只是回答问题，而是需要规划、调用工具、观察环境、修正行动时，它应该如何学习？**

## 从 LLM 到 Agent

普通 LLM 的典型输入输出形式是：

```text
用户问题 → 模型回答
```

但 Agent 的交互形式更复杂：

```text
任务目标 → 规划 → 调用工具 → 观察结果 → 调整策略 → 完成任务
```

这意味着智能体面对的不是一次性的文本生成，而是一个多步决策过程。它需要决定：

1. 什么时候直接回答；
2. 什么时候调用工具；
3. 调用哪个工具；
4. 如何根据工具返回结果修正下一步；
5. 什么时候停止行动并给出最终答案。

这类问题天然接近强化学习中的序列决策问题。

## 什么是 Agentic RL？

Agentic RL 可以理解为：**面向智能体行为的强化学习训练方法**。

它不是只奖励一句回答写得好不好，而是更关注整个任务轨迹是否成功。

一个 Agent 的轨迹可能长这样：

```text
Observation 1 → Action 1 → Observation 2 → Action 2 → ... → Final Answer
```

在这个过程中，模型可能会：

- 检索文档；
- 查询数据库；
- 调用代码解释器；
- 使用浏览器；
- 写入文件；
- 反思前一步是否出错；
- 重新规划任务路径。

Agentic RL 的目标就是让模型学会在这些复杂步骤中做出更好的选择。

## 为什么只靠 SFT 不够？

SFT，也就是监督微调，通常依赖人类写好的示范数据。

它擅长教模型：

```text
看到这种输入，应该模仿这种输出。
```

但在智能体任务中，只模仿示范有几个问题。

第一，真实任务的路径很多。一个任务可能有多种完成方式，示范数据很难覆盖所有情况。

第二，智能体会遇到动态反馈。工具调用可能失败，检索结果可能不完整，环境状态可能变化。

第三，很多错误只有执行之后才知道。比如代码写出来看似正确，但运行后才发现报错。

所以智能体不能只学“标准答案”，还要学会根据反馈调整行为。

## Agentic RL 关注的核心能力

我理解 Agentic RL 主要想提升以下几类能力。

### 1. 长程规划能力

很多任务不是一步完成的，而是需要拆解。

比如“帮我读一篇论文并生成结构化总结”，可能包含：

1. 解析论文；
2. 提取摘要、方法、实验、结论；
3. 检查图表和公式；
4. 对齐引用位置；
5. 生成总结；
6. 自我检查是否忠于原文。

Agentic RL 希望模型不仅知道每一步是什么，还能学会在不同状态下选择合适的下一步。

### 2. 工具使用能力

Agent 经常需要调用外部工具。

例如：

- 搜索工具；
- 文件读取工具；
- 数据库查询工具；
- Python 执行环境；
- 浏览器；
- 邮件和日历 API。

一个好的智能体不仅要知道工具存在，还要知道：

```text
什么时候用工具，什么时候不用工具。
```

过度调用工具会浪费资源，不调用工具又可能产生幻觉。

### 3. 反思和纠错能力

智能体执行任务时很容易出错。

比如检索到了无关文档、代码运行失败、计算结果不对、回答没有引用来源。

Agentic RL 可以把这些反馈纳入训练，让模型学习：

```text
发现错误 → 定位原因 → 修正行动
```

这也是 Reflection、Self-Correction 和 Agentic RL 经常一起出现的原因。

### 4. 任务完成度优化

普通回答可能只看语言质量，但 Agentic RL 更看最终任务是否完成。

比如一个客服智能体，最终指标可能不是“回复写得好不好”，而是：

- 是否正确理解用户诉求；
- 是否查到了订单信息；
- 是否符合退款政策；
- 是否把有争议的情况交给人工；
- 是否避免误操作。

这类目标更适合用任务级奖励来优化。

## 奖励信号从哪里来？

Agentic RL 的难点之一是奖励设计。

可能的奖励来源包括：

### 规则奖励

例如：

```text
代码是否运行通过
答案是否包含引用
工具调用格式是否正确
是否在预算内完成任务
```

这种奖励比较清晰，但容易覆盖不全。

### 人类反馈

人类可以评价一条轨迹是否合理。

优点是质量高，缺点是成本高。

### 环境反馈

很多任务可以从环境中直接拿到结果。

比如：

```text
代码任务：测试是否通过
网页任务：按钮是否点击成功
问答任务：答案是否命中标准答案
检索任务：是否找到相关证据
```

这种反馈很适合智能体训练。

### 模型评审

还可以让另一个模型评价轨迹质量。

这类方法成本较低，但需要小心评审模型本身的偏差。

## Agentic RL 和 RAG 的关系

RAG 解决的是：

```text
模型如何获得外部知识
```

Agentic RL 解决的是：

```text
模型如何决定行动策略
```

在论文阅读智能体中，RAG 可以负责找到相关段落，而 Agentic RL 可以让智能体学会：

1. 什么时候检索；
2. 检索哪些问题；
3. 如何判断检索结果是否足够；
4. 什么时候需要再次检索；
5. 如何基于证据生成回答。

所以 RAG 更像知识获取机制，Agentic RL 更像行为优化机制。

## 一个简单的 Agentic RL 任务例子

假设任务是：

```text
阅读一篇论文，回答它的方法创新点是什么，并给出原文依据。
```

一个普通模型可能直接回答。

一个 RAG Agent 可能会先检索相关段落再回答。

一个经过 Agentic RL 优化的智能体，理想情况下会：

1. 先判断问题类型；
2. 检索 introduction 和 method 部分；
3. 提取候选创新点；
4. 再检索实验部分验证这些创新点是否真的被实验证明；
5. 生成答案；
6. 检查每个结论是否有证据支撑；
7. 如果证据不足，主动说明不确定性。

这里的关键不是单次回答，而是完整行为链条的质量。

## 目前的挑战

Agentic RL 也有很多困难。

### 训练成本高

智能体任务需要和环境交互，训练成本通常比普通文本训练更高。

### 奖励容易被投机利用

如果奖励设计不合理，模型可能学会钻空子。

比如只为了通过格式检查，而不真正解决问题。

### 长程信用分配困难

一个任务失败，可能是第一步规划错了，也可能是最后一步总结错了。

如何判断是哪一步导致最终失败，是强化学习里的经典难题。

### 安全边界更复杂

Agent 可以调用工具、写文件、发邮件、操作网页，所以错误行为的影响比普通聊天更大。

因此 Agentic RL 不仅要优化能力，还要加入权限控制、审计机制和人工兜底。

## 我的理解

Agentic RL 的意义在于：它把大模型从“文本生成器”进一步推向“任务执行者”。

普通 LLM 更像一个会回答问题的大脑。

Agent 则像一个能使用工具、观察世界、逐步完成任务的系统。

而 Agentic RL 试图解决的问题是：

```text
如何让这个系统在真实反馈中变得更会做事。
```

对我来说，Agentic RL 很适合作为后续学习 Agent 的主线之一。因为它连接了几个重要方向：

- 强化学习；
- 工具调用；
- RAG；
- 长程规划；
- 自我反思；
- 任务级评估；
- 安全与对齐。

后续如果继续深入，可以进一步学习具体算法、训练框架、环境构造和评测基准。
